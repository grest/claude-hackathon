02 / 08 The Problem
problem statement

Same company. Same month.
Same underlying data.
Four answers.

Churn is the canonical contested metric. Logo vs. revenue. Downgrades counted or not. When the clock starts. How long the reactivation window is. Whether pauses count.

A finance director, a CS ops manager, and the analyst who built the old dashboards will each calculate it differently — and each will be defensible.

That is the whole point. The problem isn't bad math. It's missing governance. The business has forty dashboards and zero shared vocabulary.

05 / 08 The Engine
the engine

Every number
carries its paperwork.

The engine is boring on purpose. One calculation path, five configurations. Definitions are data, not code branches.

Every ChurnResult is annotated with engine_version, an exclusion_reason when something isn't counted, and a late_notification_flag when the clock was started suspiciously late. Multi-year contracts churn at renewal date, not at notification.

When a VP asks "why is this 4.2% and not 3.1%?", the answer is a trace, not an argument. The boundary cases are in the definition doc. Twelve of them. Signed off.

"The four legacy definitions are modeled in code, not prose. This is why the reconciliation table is honest." — ADR-001

@dataclass class ChurnResult: account_id: str is_churn: bool mrr_lost: float churn_date: date days_to_contract_end: int late_notification_flag: bool churn_reversed: bool engine_version: str = "1.0" exclusion_reason: Optional[str] = None # Boundary cases — v1.0 · 2026-04-24 # · trials: excluded # · temp discounts: excluded # · win-backs ≤30d: reversed # · multi-year: date = renewal # · downgrades: delta MRR counts
